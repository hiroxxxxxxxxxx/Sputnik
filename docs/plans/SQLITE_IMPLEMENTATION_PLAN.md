## Phase 0: 仕様・方針の確定

### 0.0 「高度 2値 / 3値」の意味（用語説明）

**高度 (altitude)** は、金利サイクルに応じた運用レジームを表す。

- **定義書 (SPEC)** では **3 値**：「高高度（利下げ）・中高度（安定期）・低高度（利上げ）」で、閾値テーブルや CC 方針が 3 段階で書かれている。
- **現行コード** では **2 値**：`AltitudeRegime = "high_mid" | "low"`。`v_factor.py` や `factors_config` の閾値は `high_mid` と `low` の 2 キーで保持している。

したがって「高度は 2 値か 3 値か」は、**DB に保存する値のとりうる範囲**をどちらに合わせるかを聞いている。

- **(A) 2 値**：現コードのまま `high_mid` / `low` を保存。実装がそのまま使える。
- **(B) 3 値**：定義書に合わせて 高/中/低 を保存。コード側の閾値やロジックも 3 段階に拡張する。

**確定**: **(B) 3 値**。定義書に合わせて実装を変更する（`AltitudeRegime` および閾値テーブルを 高/中/低 に拡張）。  
※ 高度の API 集約（altitude を build_signal_bundle 引数からレジーム設定へ移行）は `docs/spec/TODO.md` 項目 4 参照。P2-0 と合わせて実施が望ましい。

### 0.3 Mode（制御系）— 承認・執行ロック詳細は後フェーズ

#### 「執行ロック」とは

**執行ロック** は、**発注・プロトコル執行を一時的に止めるための安全スイッチ**（安全装置）を指す。

- **ON（ロック中）**: システムは注文を出さない。プロトコルで「執行」まで行う処理も、実際の発注だけ行わない、またはプロトコル実行自体をスキップする、といった制御を想定。
- **OFF**: 通常どおり、承認フローや AP モードに従って執行する。

定義書には「執行ロック」という見出しはないが、データ仕様の Mode カテゴリに「執行ロック」を入れていたため計画に含めた。**現コードには未実装**なので、先行フェーズでは DB に「ロック中かどうか」のフラグだけ持っておき、実際にロックを参照する制御は注文・執行実装時に追加する想定でよい。

| # | 質問 | 回答 | メモ |
|---|------|------|------|
| **M1** | **承認フラグ** の永続化 | **後フェーズ** | 注文・執行機能が未実装のため、承認フラグも**実装後に検討**。要計画（後述）。先行では APモード・執行ロックのみ永続化。 |
| **M2** | 執行ロックに「理由・開始日時」を残すか？ | **後フェーズ** | 注文系と合わせてログ実装時に検討。先行では **execution_lock フラグのみ** 永続化する。 |

### Phase 0 まとめ（確定事項）

- **State**: 高度は **3 値**（高/中/低）。定義書に合わせてコードも拡張。高度変更履歴は持つ。直近スロットルは **運用開始時に各 part の先物枚数（初期値）を決め、ロール・リバランス時に見直し可能**とする。
- **Mode**: 先行では APモード・執行ロック（フラグのみ）。執行ロックは「発注・執行を止める安全スイッチ」。承認フラグ・執行ロックの理由・開始日時は **注文系・ログ実装時に後フェーズで検討**。
- **Signal**: 因子レベルは日次保存。補助計器は後フェーズで実装後に永続化。
- **ログ**: 後フェーズで検討。**B3（バックアップ）はログ実装時に非機能面としてまとめて検討**。
- **基盤**: B1 は環境変数で上書き可能、**デフォルトパスは C（`data/sputnik.db`、プロジェクトルート）**。B2 単一ライター、B4 無制限で開始。B3 は上記のとおり。**設定（config）は現状 `src/config` だが、ルートの `config/` に寄せることを推奨**。移行は別タスクで可。
- **日次スケジュール**: **0.6** 参照。`signal_daily` は NY クローズ後 1 回／日次レポート Telegram は JST 朝 7:00～8:00 頃。

## 後フェーズ計画（要計画）

以下の 3 つは、該当機能の実装状況に合わせて別フェーズで計画・実装する。

### 後フェーズ A: 補助計器の永続化

- **前提**: Realized Skew, Theta率、その他 SPEC 2-3 の補助計器の**算出コードが実装されている**こと。
- **内容**: (1) `signal_daily` に補助計器用カラムを追加するマイグレーション (2) 日次バッチで算出結果を 1 日 1 回同じテーブルに書き込む (3) 既存の因子レベル日次と as_of で紐づく。
- **計画で決めること**: 保存する補助計器の一覧、カラム名、算出タイミング（どのバッチのいつか）。

### 後フェーズ B: 承認フラグの永続化

- **前提**: 注文・執行機能が実装され、Telegram 承認フローが動いていること。
- **内容**: (1) `mode` に承認待ちフラグ・待機中 signal 要約（任意）を追加するマイグレーション (2) 承認待ち開始・承認/却下・再起動時のクリア or 復元のルールを仕様化 (3) Cockpit の承認状態変更時に DB を更新する。
- **計画で決めること**: 承認フラグの意味（待ち中かどうかのみか、要約も持つか）、再起動時の扱い（クリアするか復元するか）。

### 後フェーズ C: ログ（ADVISORY・執行ログ）

- **前提**: ADVISORY/CAUTION/WARNING の発生箇所と、プロトコル実行・承認/却下の呼び出し箇所が明確になっていること。
- **内容**: (1) `advisory_log` / `execution_log` テーブル作成 (2) 種別・理由・日時・effective_level 等の項目を決める (3) 発生時・実行時に insert (4) 保持期間を決め、古い行の削除 or アーカイブのジョブを検討。
- **計画で決めること**: 保持期間（L1, L2）、Telegram 承認/却下を記録するか（L3）、削除ポリシー。
