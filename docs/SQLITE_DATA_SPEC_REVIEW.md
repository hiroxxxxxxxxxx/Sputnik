# SQLite データ仕様 検証・改善案

データ管理一覧の検証結果。漏れ・改善案・実装上の注意をまとめる。

---

## 1. カテゴリ別検証

### 市場データ（価格, IV, 金利等）

| 項目 | 判定 | 備考 |
|------|------|------|
| 都度取得・保存しない | ✅ 妥当 | API から再取得可能。SPEC 3-1 の Raw Data と一致。 |

**漏れ**: なし。

---

### Signal（補助計器）— SQLite 日次

| 項目 | 判定 | 備考 |
|------|------|------|
| V, U, S 因子の計算値 | ✅ 妥当 | 月次レビュー・ボラ歪み・インカム効率の根拠。 |
| Realized Skew, Theta率 | ✅ 妥当 | SPEC 2-3 の「機体診断」「エンジン効率」と一致。【漏れ修正】済み。 |

**漏れ・改善案**

1. **日付キーを明示**  
   必ず `as_of`（営業日）を主キーまたは複合キーに含める。銘柄別がある場合は `(as_of, symbol)`。

2. **P, C, R, T のレベル or 入力値**  
   月次で「なぜその Effective になったか」をトレースするなら、V/U/S に加え **ICL を構成する P,V,T および SCL,LCL を構成する C,R,U,S のレベル（0/1/2）または入力シグナル値**を日次で保存するとよい。現仕様は「補助計器」に特化しているので、必要なら「Signal 拡張」として検討。

3. **補助計器の網羅**  
   SPEC 2-3 にはほかに **Basis Stress, Corr20, Portfolio Vega, Current DD, Sharpe Ratio** がある。月次レビューで使うなら、Realized Skew・Theta率と同様に「日次で保存する補助計器」の対象に含めるかどうか決めておく。

4. **Realized Skew / Theta率の算出**  
   現コードでは未実装の可能性あり。保存するなら算出ロジック（IV_Put, IV_ATM, Total Theta, MM の取得元）と保存タイミング（日次バッチのいつか）を仕様に書く。

---

### State（飛行状態）— SQLite

| 項目 | 判定 | 備考 |
|------|------|------|
| Effective Level | ✅ 妥当 | 再起動復旧・「現在の段位」に必要。 |
| 高度 | ✅ 妥当 | 6ヶ月維持ルール・閾値テーブル切り替えの根拠。 |
| 高度変更日 | ✅ 妥当 | 6ヶ月未満は変更しない制約の判定に必須。 |

**漏れ・改善案**

1. **高度の値の定義**  
   コードでは `AltitudeRegime = "high_mid" | "low"`（2値）。定義書は「高・中・低」の3値。仕様に **保存する値のとりうる範囲**（例: `high_mid`, `low` または 高/中/低）を明記する。

2. **高度変更の監査**  
   「高度変更日」だけでなく **変更前の高度・変更後の高度** を 1 行で残すと、6ヶ月ルールの監査がしやすい（例: `altitude_changes` テーブルに `changed_at, from_altitude, to_altitude`）。

3. **直近スロットルモード**  
   再起動後、`pulse()` で再取得する前提なら必須ではないが、「直近の Boost/Cruise/Emergency」を State に持っておくと復旧表示が早くなる。任意。

4. **ICL/SCL/LCL の直近値**  
   トレース用に「直近の ICL, SCL, LCL」（銘柄別は ICL のみ）を State に持つかは、Signal 日次で代替できるなら不要。重ねて持つ場合は「State は直近1件、時系列は Signal 日次」と役割を分けるとよい。

---

### Mode（制御系）— SQLite

| 項目 | 判定 | 備考 |
|------|------|------|
| APモード | ✅ 妥当 | Manual / Semi / Full。ユーザー意志の永続化。 |
| 承認フラグ | ⚠️ 要定義 | 「何の承認か」を仕様で一言あるとよい（例: 直近の Telegram 承認が granted か、または「承認待ち中か」のフラグ）。 |
| 執行ロック | ✅ 妥当 | 安全装置。 |

**漏れ・改善案**

1. **承認フラグの意味**  
   STATEFUL_AUDIT では「承認待ちは再起動で失う」とある。永続化するなら **承認待ち中かどうか** および **待機中の signal 要約（reason, throttle_level 等）** を保存するかどうか決める。保存する場合は、再起動後に「承認待ちが残っている」と復元できるが、Telegram 側のセッションは切れているので UX をどうするか（例: 再起動したら承認待ちはクリアする、など）を決める。

2. **執行ロックの監査**  
   ロックの **理由・開始日時**（任意で解除日時）を残すと、トラブル時の確認に役立つ。

3. **設定の保存タイミング**  
   APモードや承認フラグを「いつ書き込むか」（変更時のみ vs 定期的スナップショット）を決める。変更時のみで十分なことが多い。

---

### ログ（監査）— SQLite

| 項目 | 判定 | 備考 |
|------|------|------|
| ADVISORY 発生履歴 | ✅ 妥当 | 補助計器逸脱の記録・Telegram 通知の証跡。 |
| 執行ログ | ✅ 妥当 | どのプロトコルをいつ実行したかの証跡。 |

**漏れ・改善案**

1. **ADVISORY の種別**  
   **ADVISORY / CAUTION / WARNING** を区別して保存する（SPEC 2-4）。WARNING は Emergency 発動と一体なので、**Emergency プロトコル発動日時・トリガー理由（例: LCL=2）** を同じテーブルまたは関連テーブルに明示すると「証跡」として分かりやすい。

2. **保存する項目の例**  
   - 種別（ADVISORY/CAUTION/WARNING）  
   - 理由（補助計器名 or Effective Level 等）  
   - 発生日時  
   - 関連する Effective Level や主要因子（任意）  
   - Telegram 通知の有無・成否（任意）

3. **執行ログの項目例**  
   - プロトコル名（Emergency, BoosterIgnition, …）  
   - 実行日時  
   - 入力の要約（signal.reason, throttle_level 等）  
   - 結果（成功/失敗、注文 ID やエラー内容は別テーブルやログメッセージで）  
   - Telegram 承認の「承認/却下」も記録すると監査で便利。

4. **保持期間**  
   無制限だと DB が膨らむ。例: **ADVISORY 系 1 年、執行ログ 2 年**。定期的に古い行を削除するか、アーカイブ用に別 DB に退避するポリシーを決める。

---

### 戦略定数（各種閾値, Base Density）— 設定ファイル

| 項目 | 判定 | 備考 |
|------|------|------|
| 設定ファイルで管理 | ✅ 妥当 | 憲章に基づき書き換えを制御。 |

**改善案**

1. **バージョン管理**  
   設定は git で管理し、**誰がいつどの変更をしたか** が分かるようにする。

2. **本番の変更ログ**  
   デプロイ履歴や「本番で設定を変更した日時・変更内容」を簡易ログ（ファイルまたは DB の 1 テーブル）に残すと、「憲章に基づき管理した」ことを示しやすい。

## 2. 共通・実装上の注意

### スキーマとマイグレーション

- 初回導入時から **スキーマバージョン**（例: 1, 2, …）を DB に持つ。
- テーブル追加・カラム追加は **マイグレーションスクリプト** で行い、バージョンごとに 1 本のスクリプトにすると運用しやすい。

### 書き込みの単一化

- SQLite は **単一プロセス・単一ライター** を推奨。複数コンテナや複数プロセスから同時に書く場合は、キューを経由するか、書き込み担当を 1 プロセスにまとめる。

### バックアップと保持

- SQLite ファイルの **バックアップ頻度**（日次など）を決める。
- Signal 日次は **保持期間**（例: 2 年）を決め、古いデータのアーカイブまたは削除方針を書いておく。

### 日次 vs リアルタイム

- **Signal（日次）**: 日次バッチで 1 日 1 回書き込む想定。
- **State / Mode / ログ**: イベント発生時または状態変更時に即時書き込む想定。  
  書き込みタイミングを分けて設計すると、日次バッチの負荷とリアルタイム応答を両立しやすい。

## 3. まとめ（推奨アクション）

| 優先度 | 内容 |
|--------|------|
| 必須 | 各テーブルに **as_of または occurred_at** を明示。ログの **保持期間** を決める。 |
| 推奨 | State の **高度の値の定義**（high_mid/low 等）を仕様に明記。**高度変更履歴**（前後値）を保存。 |
| 推奨 | ログで **ADVISORY/CAUTION/WARNING の種別** と **Emergency 発動の明示**。執行ログに **承認/却下** を記録。 |
| 任意 | Signal に **P,C,R,T のレベル** を追加。補助計器に **Current DD, Portfolio Vega** 等を追加。Mode の **承認待ちの永続化** と再起動時の扱いを仕様化。 |
